## 目标
- 在生成消息数据时预先计算文本布局宽高与末行信息，减少 View 的 onMeasure/StaticLayout 构建开销。
- 发送消息后、加入列表前完成计算，并把结果写入 model 字段供渲染复用。

## 方案概述
- 新增一个文本布局“预计算包”，包含：内容宽度、高度、末行宽度、末行基线、为时间预留的右侧宽度与静态布局块集合。
- 在 Activity/Adapter 层获取容器可用宽度；调用预计算器生成该包并写入 TgMessageItem.Text。
- Cell/ContentView 绑定时直接使用预计算包进行测量与绘制，避免重复构建 StaticLayout。

## 数据结构
- 新增数据类 TgTextLayoutPack：
  - blocks: List<TgTextLayoutBlock>
  - contentWidth: Int
  - height: Int
  - lastLineWidth: Int
  - lastLineBaseline: Float?
  - reservedRight: Int
  - inlineTimeWithText: Boolean
  - computedForWidth: Int（用于校验是否与当前可用宽度一致）
- 扩展 TgMessageItem.Text 增加字段 layoutPack: TgTextLayoutPack?

## 预计算流程
1. 获取密度与画笔：TgTheme.init(resources.displayMetrics.density)，使用 chatMsgTextPaint、chatTimePaintOut/In。
2. 计算容器最大气泡宽：containerWidth * maxBubbleWidthRatio - paddingStart/End。
3. 计算时间宽度（依据 fromMe 选择时间画笔）；判断是否需要行内时间并得出 reservedRight。
4. 按 availableWidth = maxBubbleWidth - reservedRight 构建 StaticLayout；用 TgTextLayoutBuilder 切块 blocks。
5. 统计 contentWidth 与总高度、末行宽度与基线；封装为 TgTextLayoutPack。
6. 将 pack 写入 TgMessageItem.Text.layoutPack。

## 适配器与 Cell 改动
- TextVH.bind：优先取 item.layoutPack，传给 TgTextMessageCell/TextContentView（新增 setLayoutPack(pack)）。
- TextContentView：
  - 若有 pack 且 computedForWidth 与当前可用宽度一致，onMeasure 直接返回 pack 的 contentWidth/height，onDraw 直接绘制 pack.blocks。
  - 若没有或宽度不匹配，按现有逻辑回退为动态计算。

## 尺寸来源
- 在 TgTextChatActivity 中，等待 RecyclerView 完成测量后记录 recyclerWidth；可用宽度参考当前实现的 0.72 比例与左右 padding。
- 发送消息（addMessage）或批量种子（seedMessages）时调用预计算器并写入 model。

## 线程与性能
- 预计算可在后台线程（Coroutine Dispatchers.Default 或 HandlerThread）执行；StaticLayout 构建不依赖 View 线程，安全性高。
- 主线程仅进行列表提交；若后台结果稍后返回，用 Diff 更新该消息的 layoutPack（保持 id 不变）。

## 回退机制
- 屏幕旋转/容器宽度变化时，computedForWidth 不匹配则触发重新计算或走现有动态测量路径。
- 当存在“额外区块”（引用/翻译/点赞）时，inlineTimeWithText 可能退化为换行时间行；预计算器需按与 BaseTgMessageCell 一致的规则处理。

## 验证
- 打开滚动列表并批量发送长文本，观察掉帧与卡顿是否降低。
- 对比 Systrace/Log：onMeasure 次数与 StaticLayout 构建次数明显减少。
- 测试不同方向（fromMe）与是否存在额外区块的时间布局正确性。
